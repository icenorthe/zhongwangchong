# 前端重构交接文档

更新时间：2026-03-23

## 1. 文档目的

这份文档是给接手本项目的高级前端工程师用的，目标是回答三个问题：

1. 这个项目现在到底已经做到了什么。
2. 哪些能力已经由后端、数据库、本地执行端、支付链路和部署链路承担，不需要前端重做业务。
3. 前端如果整体重构，应该以哪些文件、接口、数据结构、业务规则为准。

结论先说：

- 当前项目不是“只有一个静态页”，而是一个已经具备账号、余额、充值、下单、订单管理、后台管理、Agent 在线状态、实时插座状态、失败退款等能力的完整系统。
- 真正欠缺的是前端工程化、组件化、可维护性和视觉层重做。
- 现有前端已经把主要业务闭环跑通，只是实现方式仍是单文件 HTML + 原生 JS。

## 2. 一句话概况

这是一个“校园电动车充电代下单 / 自助下单”系统，整体架构是：

- 云端 Web 站点负责用户注册登录、余额、充值、订单、后台、Agent 状态、实时状态聚合。
- 本地 Windows 电脑负责实际执行充电动作，并把结果、心跳、实时快照、消费记录回传云端。
- 当前生产主干后端是一个 FastAPI 应用，前端是 3 个单文件页面。

仓库里还保留着“微信 RPA 自动点击”的说明和素材，但当前主线代码里的 `services/local_charge_runner.py` 已经是直接调用官方 HTTP 接口 `mp_balancePay.action` 的执行器，读取的是 `config/charge_api_config.json`。

也就是说：

- 当前前端交接请以“HTTP 执行器 + 云端 Agent + 余额/充值/订单系统”这条主线理解。
- 历史 RPA 资料可以视为备用方案，不是当前前端重构的阻塞项。

## 5. 系统架构

### 5.1 云端模式

这是当前最重要的主线模式。

流程如下：

1. 用户访问 PythonAnywhere 站点。
2. `services/pythonanywhere_app.py` 加载 `config/pythonanywhere_secrets.json`，并强制 `WORKER_ENABLED=0`。
3. 用户注册、登录、充值、下单，所有业务都写入云端 SQLite。
4. 新订单先进入 `PENDING`。
5. 本地电脑运行 `services/cloud_agent.py`，持续上报心跳并抢单。
6. Agent 调用 `services/local_charge_runner.py` 执行真实充电动作。
7. Agent 把结果回写为 `SUCCESS` 或 `FAILED`。
8. Agent 还会持续把插座快照和消费记录推回云端，供前端查看实时状态和结束明细。

### 5.2 本地 worker 模式

这是一条可用但不是云端主站主线的模式。

流程如下：

1. 本地运行 `services/mobile_charge_server.py`。
2. 当 `WORKER_ENABLED=1` 时，FastAPI 内部 worker 线程会直接处理订单。
3. worker 按 `config/gateway_config.json` 调用目标执行端。
4. 当前默认配置里，`gateway_config.json` 指向 `http://127.0.0.1:9000/api/start-charge`。
5. `services/local_bridge_api.py` 再调用 `services/local_charge_runner.py`。

### 5.3 支付中转 / 实时代理

为绕开 PythonAnywhere 免费版的一些外部请求限制，项目还预留了一个 Cloudflare Worker：

| 路径 | 作用 |
| --- | --- |
| `deploy/cloudflare_payment_bridge/src/index.mjs` | 充值支付中转、支付回调、实时站点状态代理 |
| `deploy/cloudflare_payment_bridge/README.md` | Worker 部署说明 |

这层不是强制，但当前前端已经适配：

- `payment_bridge_url`
- `socket_overview_bridge_url`

## 8. 关键接口分组

### 8.1 鉴权约定

| 身份 | 请求头 |
| --- | --- |
| 普通用户 | `Authorization: Bearer <session_token>` |
| 管理员 | `X-Admin-Token: <admin_token>` |
| 后台密码 | `X-Admin-Password: <admin_password>` |
| Agent | `X-Agent-Token: <agent_token>` |

说明：

- 普通用户 token 是服务端 session token，保存在 `user_sessions` 表，不是 JWT。

## 10. 当前业务规则

### 10.1 用户与风控

- 注册手机号必须通过 `normalize_phone`，最终按 11 位数字处理。
- 新用户注册默认不赠送免服务费次数。
- 首页访问会写入 `site_visitors`，并设置 visitor cookie。
- 注册时会按 visitor 和 IP 做限流与防薅。

### 10.2 服务费与时长

当前默认业务参数：

- `SERVICE_FEE_YUAN = 0.5`
- `MINUTES_PER_YUAN = 207`
- 估算充电时长 = `amount_yuan * 207`
- 订单总价 = `充电金额 + 服务费`
- 若本次使用了免服务费次数，则服务费为 0

### 10.3 充值赠送规则

当前代码中的充值赠送规则：

- 首笔成功充值 `>= 5` 元，赠送 `1` 次免服务费
- 单笔充值 `>= 10/20/30` 元，分别赠送 `1/3/5` 次免服务费
- 首充奖励和单笔档位不叠加，按更高档位发放
- 赠送内容是免服务费次数，不折算为余额到账金额

### 10.4 订单状态机

```text
PENDING -> PROCESSING -> SUCCESS
PENDING -> PROCESSING -> FAILED
FAILED -> retry -> PENDING
SUCCESS -> 匹配到空载自动结束 -> FAILED（自动退款）
```

补充说明：

- 云端模式下，`claim` 之后订单会从 `PENDING` 改成 `PROCESSING`
- 成功后进入 `SUCCESS`
- 失败后进入 `FAILED`
- 后台可以把失败单重试回 `PENDING`

### 10.5 余额扣费规则

`PAYMENT_MODE=balance` 时：

1. 创建订单时先预扣余额。
2. 若订单失败，自动退款。
3. 若失败单重试，会再次预留扣款。
4. 若下单时还有免服务费次数，会先预占一次，失败后归还。

### 10.6 订单限流与保护

系统会按“设备编码 + 插座号”做保护：

- 最近刚提交过订单时，短时间内禁止重复提交
- 近期失败次数过多时，临时禁止提交
- 近期出现“空载 / 未插好”失败时，临时禁止提交

### 10.7 空载自动退款

这是一个已经落地的自动化规则：

- 当订单表面上已经 `SUCCESS`
- 但后续从官方消费记录里匹配到“空载自动结束”
- 且该单是余额支付、已经扣款、尚未退款

系统会自动：

- 把订单修正为 `FAILED`
- 自动退回余额
- 给出更明确的失败文案

### 10.8 站点规则

- 隐藏站点号不会出现在前端列表中
- `socket_remap` 允许“用户选 4 号，实际走 7 号”这类映射
- 如果站点只有占位配置、没有 `device_code`，则允许展示但不允许下单
- 当前 76 个可见站点都已具备真实 `device_code`

## 15. 对前端重构的明确建议

### 15.1 可以放心不动的层

高级前端接手时，可以默认先不动这些层：

- 用户、订单、充值、余额、后台、Agent、实时状态的后端接口
- SQLite 表结构
- 云端 Agent 抢单和完单链路
- 失败退款和空载自动退款规则
- 充值审核和充值赠送规则

### 15.2 前端重构时应保留的业务能力

以下能力不建议删减，因为它们都已经在现有页面中落地：

- 游客可看站点状态
- 登录注册
- 余额展示
- 免服务费次数展示
- 站点分区浏览
- 实时插座状态与倒计时
- 下单前实时检查
- 我的订单
- 订单结束明细
- 充值申请与充值记录
- 钱包流水
- 后台订单与充值审核
- 后台用户管理

### 15.3 建议的重构边界

建议把重构目标拆成 3 个前端应用，而不是继续维护 3 个大 HTML 文件：

1. 用户端 App，对应 `/`
2. 订单后台 App，对应 `/admin/orders`
3. 用户后台 App，对应 `/admin/users`

### 15.4 建议优先做的工程化改造

- 组件化：站点卡片、插座卡片、订单卡片、充值记录、后台表格、状态徽标
- API 层收口：统一处理 `Authorization`、管理员 headers、错误码和重试
- 状态管理：登录态、服务状态、站点状态、轮询任务统一收口
- 页面模块化：不要再把全部逻辑塞进一个脚本块
- 轮询策略标准化：订单轮询、充值轮询、后台刷新、倒计时更新解耦
- 类型化：建议基于当前响应结构补完整 TS 类型

### 15.5 一个非常现实的约束

当前后端默认按“同域静态页面 + 同域 REST”来设计。

如果前端想独立成单独域名，需要额外处理：

- CORS
- 登录态传递
- Worker 域名与主站域名之间的授权头透传

所以若想最快上线，仍建议：

- 前端编译产物部署回 `assets/web`
- 继续由 FastAPI 同域提供页面和接口

## 16. 当前已知问题 / 风险提示

这部分不是说系统不能用，而是给重构时避坑。

### 16.1 前端层面的债务

- 当前用户端和后台端都是单文件原生 JS，后续维护成本已经很高
- API 调用、轮询、状态处理散落在页面脚本里
- 存在部署副本文件，容易改错地方

### 16.2 文档与代码有历史错位

- 仓库里有不少早期文档仍在讲微信 GUI RPA
- 但当前实际执行器主线已经是 HTTP 接口执行
- 新前端对接时请以代码为准，不要只看旧文档

### 16.3 后端有“已支持但当前前端未重点使用”的能力

例如：

- `PAYMENT_MODE=token`
- `/api/admin/payments/issue`

这些能力后端已具备，但当前现有前端不是主入口。重构时可决定是否纳入第一期。

## 17. 给接手前端的最终判断

如果把这个项目交给一个高级前端工程师，合理的判断应该是：

- 这不是一个“前后端都没定义”的项目。
- 这是一个“后端和业务链路已经基本成型，但前端实现方式过于原始”的项目。
- 最适合的做法不是重新发明业务，而是基于现有 REST、状态机、配置和 URL 结构，把前端整体重构成可维护的应用。

换句话说，当前真正缺的不是业务定义，而是前端工程化重建。
